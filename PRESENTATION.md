## 🎯 Обзор проекта

**Telegram-бот для управления ресторанным бизнесом** — комплексное решение для автоматизации работы ресторанов, управления меню, акциями, событиями и взаимодействия с клиентами.

### Основные возможности:
- ✅ Просмотр ресторанов по регионам
- ✅ Интерактивное меню и банкетные предложения
- ✅ Система акций и событий
- ✅ Личный кабинет для ресторанов
- ✅ Централизованное управление администраторами
- ✅ Система согласования всех изменений
- ✅ Автоматические рассылки клиентам

## 🏗️ Архитектура системы

### Технологический стек:
- **Бот-фреймворк**: Aiogram 3.7.0 (асинхронный)
- **База данных**: PostgreSQL (asyncpg)
- **Хранилище состояний**: Memory Storage (FSM)
- **Интеграция**: Google Sheets для синхронизации данных
- **Обработка файлов**: PDF → изображения с кэшированием

### Модульная архитектура:
```
restobot/
├── handlers/          # Обработчики сообщений
│   ├── common.py     # Общие команды (/start)
│   ├── user.py       # Функционал для пользователей
│   ├── admin.py      # Административная панель
│   ├── restaurant.py # Личный кабинет ресторана
│   ├── approval.py   # Система согласований
│   └── promotions_events.py # Акции и события
├── database/          # Работа с БД
│   ├── queries.py    # SQL-запросы
│   └── sync.py       # Синхронизация с Google Sheets
├── utils/            # Вспомогательные функции
│   ├── keyboards.py  # Клавиатуры
│   ├── helpers.py    # Утилиты
│   └── pdf_converter.py # Конвертация PDF
├── middleware/       # Промежуточное ПО
│   └── role_check.py # Проверка ролей
└── states/           # FSM состояния
    └── bot_states.py
```

## 👥 Роли пользователей

### 1. 👤 **Пользователь (User)**
- Базовый доступ к просмотру ресторанов
- Просмотр меню, акций, событий
- Информация о бронировании и доставке

### 2. 🏢 **Ресторан (Restaurant)**
- Доступ к личному кабинету своего ресторана
- Управление меню, акциями, событиями
- Редактирование информации о ресторане
- Управление модераторами ресторана
- Все изменения отправляются на согласование

### 3. ⭐ **Администратор (Admin)**
- Полный доступ ко всем функциям
- Управление пользователями и ролями
- Согласование изменений ресторанов
- Добавление и редактирование ресторанов
- Отправка рассылок всем пользователям
- Доступ к личному кабинету ресторана (если является модератором)

**Особенность**: Администратор может быть одновременно модератором ресторанов, но сохраняет статус администратора.

## 🏢 Личный кабинет ресторана

### Доступ к кабинету
- Рестораны могут управлять только своими заведениями
- Администраторы могут управлять любыми ресторанами
- Поддержка множественных ресторанов для одного пользователя

### Управление информацией о ресторане

#### ✏️ Редактирование данных
- **Название ресторана**
- **Адрес**
- **Информация о бронировании**
- **Условия доставки**
- **Регион**

Все изменения отправляются администраторам на согласование.

### Управление меню

#### 📋 Изменение меню
- Файловый менеджер для PDF-файлов
- Загрузка новых меню
- Замена существующих меню
- Удаление меню
- Автоматическое кэширование при загрузке
- Очистка кэша при удалении

**Процесс:**
1. Ресторан загружает новый PDF
2. Файл сохраняется во временной папке
3. Администратор получает уведомление
4. Администратор просматривает старое и новое меню
5. После одобрения файл переносится в основную папку
6. Кэш автоматически обновляется

### Управление банкетами

#### 🎉 Изменение банкетных предложений
- Аналогичный файловый менеджер
- Загрузка, замена, удаление PDF-файлов
- Полное кэширование для быстрой отправки пользователям
- Согласование через администраторов

### Управление акциями

#### 🎁 Изменение акций
- Создание новых акций (название + описание + фото опционально)
- Редактирование существующих акций
- Удаление акций
- Навигация по списку акций
- Все изменения требуют согласования администратора

**Формат акции:**
- Название
- Подробное описание
- Фотография (опционально)

### Управление событиями

#### 🎊 Изменение событий
- Создание новых событий
- Редактирование существующих событий
- Удаление событий
- Поддержка фотографий и текстовых описаний
- Согласование всех изменений

### Управление модераторами

#### ➕ Добавление модератора
1. Выбор ресторана (если их несколько)
2. Ввод Telegram `@username`
3. Автоматическое разрешение `@username` в `chat_id`
4. Уведомление нового модератора
5. Назначение роли "ресторан" (если не администратор)

#### ➖ Удаление модератора
1. Выбор ресторана
2. Выбор модератора из списка
3. Подтверждение удаления
4. Уведомление удаленного модератора

- **✅ Разбанить пользователя** — восстановление доступа
- **⭐ Добавить админа** — повышение до администратора
- **❌ Удалить админа** — понижение до пользователя

**Процесс:**
1. Выбор действия
2. Ввод Telegram `@username`
3. Автоматическое разрешение `@username`
4. Изменение роли в базе данных
5. Уведомление пользователя

### Управление ресторанами

#### ➕ Добавление ресторана
Пошаговый процесс создания:
1. **Название ресторана**
2. **Адрес**
3. **Информация о бронировании**
4. **Условия доставки**
5. **Регион**

**Особенности:**
- Кнопка "Отменить" на каждом шаге

- Выбор ресторана из списка
- Редактирование всех полей:
  - Название
  - Адрес
  - Бронирование
  - Доставка
  - Регион
- Можно изменить только нужные поля
- Все изменения отправляются на согласование

#### 🗑️ Удаление ресторана
- Выбор ресторана
- Подтверждение удаления
- Кнопка "Удалить" размещена ниже кнопки "Отменить"
- Полное удаление из базы данных с каскадным удалением связанных данных

- Разбиение длинных подписей к фото

## 🔄 Система согласования изменений

### Принцип работы

1. **Ресторан вносит изменение**
   - Любое изменение (кроме просмотра) требует согласования
   - Данные сохраняются в таблице `restaurant_approvals` или `promotion_event_approvals`
   - Для файлов создаются временные копии в папке `pending_menu/`

2. **Уведомление администраторов**
   - Все администраторы получают уведомление: "🔔 Новая заявка на согласование!"
   - Кнопка "Показать" для просмотра заявки
   - Информация о ресторане и типе изменения

3. **Просмотр и решение**
   - Администратор просматривает детали изменения
   - Для файлов: просмотр старого и нового файла
   - Решение: одобрить или отклонить

4. **Применение изменений**
   - При одобрении:
     - Данные применяются к ресторану
     - Временные файлы переносятся в основную папку
     - Старые файлы и кэш удаляются
     - Создается новый кэш для новых файлов
   - При отклонении:
     - Временные файлы удаляются
     - Ресторан получает уведомление

5. **Рассылка (опционально)**
   - После одобрения предлагается отправить рассылку
   - Администратор может отправить стандартное или отредактированное сообщение

### Типы запросов на согласование

| Тип | Поле | Описание |
|-----|------|----------|
| Ресторан | `restaurant_name` | Название ресторана |
| Ресторан | `address_nm` | Адрес |
| Ресторан | `reservation` | Информация о бронировании |
| Ресторан | `delivery` | Условия доставки |
| Ресторан | `region_nm` | Регион |
| Файл | `menu` | Меню ресторана (PDF) |
| Файл | `banquet` | Банкетные предложения (PDF) |
| Акции | `create` | Создание новой акции |
| Акции | `update` | Изменение акции |
| Акции | `delete` | Удаление акции |
| События | `create` | Создание нового события |
| События | `update` | Изменение события |
| События | `delete` | Удаление события |

## ⚙️ Технические особенности

### Кэширование PDF-файлов

#### Локальное кэширование
- Конвертация PDF в изображения при первой загрузке
- Сохранение изображений в папке `cache/`
- Хеширование файлов для определения изменений
- Автоматическая очистка кэша при удалении/замене файла

#### Telegram `file_id` кэширование
- Сохранение `file_id` после первой отправки изображения
- Использование `file_id` для мгновенной повторной отправки
- Отдельное кэширование для документов и фото

#### Предзагрузка кэша
- При запуске бота все PDF-файлы автоматически конвертируются в изображения
- Это исключает задержки при первом запросе пользователя
- Фоновый процесс не блокирует работу бота

### Оптимизация изображений

- **Размер изображений**: автоматическое уменьшение до 2000×2000 пикселей
- **Сжатие**: PNG с уровнем сжатия 9 и оптимизацией
- **Качество**: баланс между качеством и размером файла
- **Обработка ошибок**: автоматическое уменьшение при ошибках `IMAGE_PROCESS_FAILED`

### Обработка больших файлов

- **Таймауты**: увеличенные таймауты (120 секунд) для загрузки больших файлов
- **Повторные попытки**: автоматические повторные попытки при таймаутах (до 3 попыток)
- **Групповая отправка**: отправка до 10 изображений одной группой
- **Последовательная отправка**: автоматический переход на отправку по одному при ошибках

### Асинхронная архитектура

- Все операции с базой данных асинхронные
- Неблокирующая обработка файлов
- Параллельная обработка множественных запросов
- Эффективное использование ресурсов

### Интеграция с Google Sheets

- Автоматическая синхронизация данных ресторанов
- Двусторонняя синхронизация
- Настраиваемые таблицы для синхронизации
- Обработка ошибок синхронизации

## 🔒 Безопасность и надежность

### Управление доступом

#### Проверка ролей
- Middleware для проверки роли пользователя перед каждым действием
- Автоматическая блокировка неавторизованных действий
- Логирование попыток доступа

#### Изоляция данных
- Рестораны видят только свои данные
- Модераторы не могут редактировать чужие рестораны
- Администраторы имеют полный доступ

### Согласование изменений

- **Все изменения требуют согласования** (кроме чтения данных)
- Двухуровневая проверка: ресторан → администратор
- История всех изменений в базе данных
- Откат изменений при отклонении

### Обработка ошибок

- **Логирование**: все ошибки записываются в логи
- **Уведомления**: пользователи получают понятные сообщения об ошибках
- **Восстановление**: автоматическое восстановление после сбоев
- **Валидация**: проверка всех входных данных

### Резервное копирование

- Все данные в PostgreSQL (надежное хранилище)
- Синхронизация с Google Sheets (дополнительная копия)
- Версионирование изменений через систему согласований

### Производительность

- Кэширование для быстрого доступа
- Асинхронная обработка для параллелизма
- Оптимизация изображений для быстрой загрузки
- Эффективные SQL-запросы с индексами

## 📊 База данных

### Основные таблицы

#### `users`
- Хранение информации о пользователях
- Роли: `user`, `admin`, `restaurant`
- Статусы: `active`, `banned`
- Уникальный `chat_id`

#### `restaurant`
- Информация о ресторанах
- Название, адрес, регион
- Информация о бронировании и доставке
- Связи с файлами меню и банкетов

#### `user_restaurants`
- Связь пользователей с ресторанами
- Многие ко многим (один пользователь — несколько ресторанов)
- Управление модераторами ресторанов

#### `restaurant_approvals`
- Запросы на согласование изменений ресторанов
- Статусы: `pending`, `approved`, `rejected`
- История всех изменений
- Связь с пользователем-заявителем и администратором

#### `promotions`
- Акции ресторанов
- Название, описание, фото
- Статус: `pending`, `approved`, `rejected`

#### `events`
- События ресторанов
- Название, описание, фото
- Статус: `pending`, `approved`, `rejected`

#### `promotion_event_approvals`
- Запросы на согласование акций и событий
- Тип действия: `create`, `update`, `delete`
- История изменений

## 🚀 Развертывание

### Требования

- Python 3.12+
- PostgreSQL 12+
- Docker и Docker Compose (опционально)
- Poppler (для конвертации PDF)

### Конфигурация

Файл `.env`:
```env
BOT_TOKEN=your_telegram_bot_token
POSTGRES_DB=restobot_db
POSTGRES_USER=restobot_user
POSTGRES_PASSWORD=your_password
POSTGRES_HOST=localhost
POSTGRES_PORT=5432
SPREADSHEET_KEY=your_google_sheets_key
MENU_PATH=/app/rest_menu
CACHE_PATH=/app/cache
PENDING_MENU_PATH=/app/pending_menu
```

### Запуск

```bash
# Установка зависимостей
```

- ✅ Централизованное управление несколькими ресторанами

- ✅ Актуальная информация о меню, акциях, событиях

- ✅ Полный контроль над изменениями
